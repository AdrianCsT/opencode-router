# v0.3.0 — Project Memory

## Goal

Subagents stop re-exploring the codebase on every dispatch. Each project
gets a persistent memory file that gets injected before the agent
starts work, so it knows the file tree, conventions, and prior
decisions without having to discover them.

## On-disk layout

```
<project>/.opencode-router/
├── memory.md              # the canonical memory (anatomy + conventions + episodic)
├── memory-index.json      # vector chunks for retrieval (large projects only)
├── log/                   # per-dispatch entries, awaiting distillation
│   ├── 2026-05-09T14-32-00-database-optimizer.yaml
│   └── ...
├── state.json             # last build time, file mtime hash, git HEAD, log count
└── .gitignore             # ignores log/ + state.json (memory.md is shareable)
```

`.opencode-router/` is created on first dispatch in a project. The
bundled `.gitignore` keeps logs and state local; `memory.md` is
committed if the user wants. No `~/.config/opencode/projects/<hash>/`
indirection — memory lives next to the code it describes.

## Module structure

```
src/opencode_router/memory/
├── __init__.py        # public API: inject(), record(), rebuild(), clear()
├── storage.py         # path resolution, .opencode-router/ creation
├── trigger.py         # decide when to rebuild
├── anatomy.py         # build file tree + symbol index (auto)
├── conventions.py     # infer style/framework/build commands (auto)
├── episodic.py        # distill log/ → episodic section
├── log.py             # write/read YAML log entries
└── retrieval.py       # vector index for large projects
```

## Public API (Python)

```python
from opencode_router import memory

# Called by route/orchestrate before dispatch
brief = memory.inject(task: str, project: Path) -> str

# Called by route/orchestrate after dispatch
memory.record(agent: str, task: str, files_touched: list[str],
              summary: str, duration_seconds: float, project: Path) -> None

# Manual operations
memory.rebuild(project: Path, *, force: bool = False) -> dict
memory.clear(project: Path) -> None
```

## Build trigger

`trigger.should_rebuild(project)` returns True if any of:

- No memory exists yet (cold start)
- ≥10 tracked files have new mtime since `state.json` snapshot
- `git rev-parse HEAD` differs from snapshot (if git repo)
- ≥24 hours elapsed since last build
- User flagged force rebuild

Rebuilds happen **after** a dispatch completes, not before — the
current dispatch uses whatever memory exists. Async via subprocess so
the user never waits.

If git is present, use `git ls-files` to respect `.gitignore` for free.
If not, walk the directory with a built-in deny list (`node_modules`,
`.venv`, `dist`, `build`, `.env*`, `*.pem`, `*.key`, `secrets/`).

## Injection

`memory.inject(task, project)` decides per call:

| Project size | Strategy |
|---|---|
| < 3k tokens of memory | Inject verbatim, no retrieval |
| 3k–15k tokens | Always-on header (~1.5k: tree skeleton + conventions) + retrieve top-K relevant chunks |
| > 15k tokens | Header (~1.5k) + retrieve top-K only |

Top-K = 5 chunks of 256 tokens each (~1.3k tokens of retrieval).
Embedding: same `mxbai-embed-large` already in use for agent routing.

## Recording

`memory.record()` writes one YAML file to `log/`:

```yaml
agent: database-optimizer
task: "fix N+1 in users endpoint"
timestamp: 2026-05-09T14:32:00Z
duration_seconds: 45
files_touched:
  - src/users/views.py
summary: |
  Replaced N+1 with select_related('profile'). Query count: 51 → 2.
```

`files_touched` is observed from the dispatch's tool calls (Edit,
Write). `summary` is the agent's final response, verbatim — no
agent-side prompt contract. This means existing agents work without
modification.

## Distillation

When `trigger.should_rebuild()` fires AND `log/` has ≥5 new entries:

1. Read all log entries.
2. One LLM call (qwen3.5:4b, JSON mode) with a tight prompt: "From
   these task logs, extract reusable patterns, decisions, and gotchas.
   Output ≤10 bullet points."
3. Append to `memory.md` under `## Episodic`.
4. Cap section at 50 most recent bullets OR 5k tokens, whichever hits
   first. Prune oldest.
5. Move processed log files to `log/.archived/`.

## Memory.md sections

```markdown
# Project Memory
_Last built: 2026-05-09T14:00:00Z_

## Anatomy
- File tree (depth-2, files grouped by directory)
- Top-level symbols (functions, classes, exported names)
- Key dependencies (from package.json/pyproject.toml/Cargo.toml/go.mod)

## Conventions
- Language(s) detected
- Test framework
- Build/lint commands (from scripts in package.json, Makefile, etc.)
- Style hints (line length, formatter detected)

## Episodic
- Distilled patterns from logs (capped, oldest pruned)
- ...
```

## CLI additions

```text
opencode-router memory show              Print memory.md
opencode-router memory rebuild [--force] Force a rebuild
opencode-router memory clear             Delete .opencode-router/
opencode-router memory log [--tail N]    Show recent log entries
opencode-router memory inject "<task>"   Print what would be injected (debug)
```

## Router/orchestrate integration

`opencode-router route` and `orchestrate` get a new `--with-memory`
flag. When set, the JSON output includes a `memory_brief` field. The
router prompt is updated to:

1. Call `opencode-router route --top-1 --with-memory "<task>"`
2. Read JSON: `{ agent, memory_brief }`
3. Pass `memory_brief + "\n\n---\n\nTask: " + task` to the task tool's
   description
4. After task completes, call `opencode-router memory record …`

Step 4 keeps the router prompt slightly longer, but the router is
small and this is one bash call.

## Privacy / security

- Walk respects `.gitignore` when present, plus a built-in deny list:
  `.env*`, `*.pem`, `*.key`, `secrets/`, `node_modules/`, `.venv/`,
  `dist/`, `build/`.
- Anatomy stores only file paths and symbol names — no file contents.
- Retrieval chunks DO include code excerpts; document this in the
  README and let users disable retrieval (`memory.retrieval = false`
  in a future config).
- Distillation prompts are local (Ollama). No code leaves the machine
  for memory operations.

## Migration / backwards compatibility

- Zero-impact for existing users: if `.opencode-router/` doesn't
  exist, the system is dormant. Existing `route` and `orchestrate`
  commands work unchanged.
- New behavior opt-in via the router prompt update. We ship a new
  `default.md` with memory wiring; old prompts keep working without
  memory.
- No changes to `~/.config/opencode/`.

## Performance budget

| Operation | Budget |
|---|---|
| Cold rebuild (small project, <500 files) | <3s |
| Cold rebuild (large project, 5000 files) | <15s |
| `inject()` call | <50ms (retrieval) or <5ms (verbatim) |
| Distillation LLM call | <5s |
| `record()` call | <10ms (just YAML write) |

## Phased shipping

1. **Phase 1 (week 1)**: storage, trigger, anatomy, conventions.
   Memory exists, gets injected verbatim. No retrieval, no episodic.
   Ships value immediately.
2. **Phase 2 (week 2)**: log + record + episodic distillation. Memory
   grows from agent activity.
3. **Phase 3 (week 3)**: retrieval for large projects. Vector index,
   chunking, top-K.
4. **Phase 4 (week 4)**: polish, tests, docs, router prompt update,
   release.

Each phase is shippable on its own — if we stop after Phase 1, users
still get a working project map. Phase 2-4 are additive.

## Open risks

- **Distillation quality** with qwen3.5:4b. Mitigation: tight JSON-mode
  prompt, low temperature, evaluate against fixture logs in CI.
- **Memory drift** when users edit `memory.md` by hand (rebuild
  overwrites). Mitigation: rebuild merges user-added sections marked
  with a sentinel comment; document the convention.
- **Token cost on large memories**. Mitigation: the 3k/15k thresholds;
  users can run `memory clear` to reset.

## Decisions to lock before Phase 1

These shape the implementation and are worth nailing down before
writing code:

1. **Symbol extraction strategy**: regex (fast, brittle, language-by-
   language), tree-sitter (accurate, adds a dep), or skip symbols and
   only ship the file tree in Phase 1.
2. **Conventions inference scope**: which manifest files we read, in
   priority order, and what we extract from each.
3. **Distillation prompt**: exact wording, output schema, how we
   evaluate quality.
4. **Memory.md merge strategy**: how rebuild handles user edits.
